# Critic/Corrector — заметки по улучшениям

Зафиксированы при pass 10 (перенос findings_review в runner). Код не изменялся.

---

## Наблюдение 1: Chunked critic — параллельный, corrector — последовательный

**Где:** `findings_review/runner.py`, chunked critic vs corrector chunks.

**Проблема:** Critic запускает все чанки параллельно (`asyncio.gather`), а corrector обрабатывает чанки последовательно (`for chunk_ids in corrector_chunks`). При большом числе замечаний это создаёт асимметрию.

**Идея улучшения:** Параллелизм corrector по аналогии с critic (с `asyncio.Semaphore`).

**Почему не исправлено:** corrector пишет в `03_findings.json` (один файл), параллельный доступ без блокировки опасен.

---

## Наблюдение 2: `issue_ids` fallback строит F-001..N при пустых finding_id в review

**Где:** `findings_review/runner.py`, строки ~409-416.

**Проблема:** Если `reviews` не содержат `finding_id`/`id` — fallback генерирует синтетические ID `F-001..N`. Это ломает адресацию при мультичанковом corrector.

**Идея улучшения:** Явно логировать такие случаи и не запускать corrector без валидных ID.

**Почему не исправлено:** Логика существовала до pass 10; изменение повлияет на поведение corrector в production — требует отдельного pass с тестированием.

---

## Наблюдение 3: Corrector восстанавливает `03_findings_review.json` целиком после чанков

**Где:** `findings_review/runner.py`, ~строка 526-529.

**Проблема:** При chunked corrector полный `review_data` записывается обратно в `03_findings_review.json` — т.е. chunk-версии (урезанные) перезаписывают друг друга в процессе, и только в конце восстанавливается полный. Это не атомарно, при сбое между чанками файл будет в неконсистентном состоянии.

**Идея улучшения:** Писать промежуточные чанки во временные файлы и атомарно заменять только в конце.

**Почему не исправлено:** Риск изменить поведение в corner case при crash. Требует отдельного pass.

---

*Следующий шаг: улучшения применять только после полной стабилизации findings_review/runner.py в production.*
