# Таксономия EOM-аудита ГРЩ v3 (финальная)

**Источник:** 74 записи эталона + ответы эксперта (2026-04-11)
**Область применения:** аудит раздела ГРЩ (Главный Распределительный Щит)
**Исключено из scope:** заземление/молниезащита (раздел ЭГ), УЗДП, квартирные щиты, УЗО-селективность

---

## Scope аудита ГРЩ

### В scope
- Кабельные линии между ГРЩ и ВРУ (сечения, марки, количество жил)
- Автоматы отходящих линий ГРЩ (марки, номиналы, расцепители)
- Спецификации оборудования ГРЩ, ЩСН, ЩУ
- Трансформаторы тока (класс точности, коэффициенты)
- Примечания к листам ГРЩ (требования к оборудованию)
- Категория надёжности электроснабжения (из общих данных)
- Компенсация реактивной мощности (УКРМ) — желательное
- Расчётные таблицы нагрузок (арифметика)
- Однолинейная схема ГРЩ

### Вне scope
- Заземление и молниезащита → отдельный раздел ЭГ
- УЗДП квартирных щитов → раздел ЭМ
- Селективность УЗО в группах → раздел ЭМ
- Соответствие ТП-ГРЩ → внешний раздел электроснабжения

Это важное сужение: **модели не надо учить проверять то что вне ГРЩ**. Если находка относится к другому разделу — она автоматически **вне scope**.

## Классы ошибок (приоритизированные)

### Priority 1 — обязательные для прототипа

Эти три класса покрывают **все 5 definite KEEPs** эталона.

#### P1.1 — identity_uniqueness (Группа 2)
**Покрывает: #2 (ЩУ-2/Т перепутаны), #5 (дубль М-1.5)**

**Суть:** нарушение уникальности или перепутанность идентификаторов/позиций в пределах одного документа.

**Подвиды:**
- `duplicate_line_marker` — один маркер линии (M-1.5) использован для разных назначений на одной схеме
- `swapped_descriptions` — в спецификации позиция A описана как B и наоборот (ЩУ-2/Т vs ЩУ-12/Т)
- `duplicate_breaker_position` — одинаковые QF-позиции для разного оборудования

**Как детектировать:**
```python
# duplicate_line_marker
for marker, mentions_on_sheet in group_by_sheet(line_markers):
    destinations = set(m.destination for m in mentions_on_sheet)
    if len(destinations) > 1:
        yield candidate(type="duplicate_line_marker", marker=marker, destinations=list(destinations))

# swapped_descriptions — сложнее, нужен semantic check
# Пример: если description для "ЩУ-2/Т" упоминает "12 счётчиков"
# а description для "ЩУ-12/Т" упоминает "2 счётчика" → flip detected
```

**Вес в эталоне:** 40% (2 из 5 KEEPs)

---

#### P1.2 — cable_attribute_mismatch (Группа 1)
**Покрывает: #7 (нейтраль 150 vs 120), #18 (ВРУ-1 1 PE vs 2 PE)**

**Суть:** атрибуты одной и той же кабельной линии различаются между источниками (однолинейная схема ↔ узлы раскладки ↔ спецификация).

**Важное ограничение:** класс работает **только если extractor умеет правильно сопоставить кабель** в разных местах. 5 из 17 записей эталона в этом классе — FP именно из-за неправильного сопоставления (#20: М-1.5 120 vs 240 — на самом деле это два разных кабеля).

**Ключ сопоставления:** `line_id` (M-1.1, M-2.3 и т.д.) — должен быть точно одинаковым.

**Поля для сравнения:**
- `mark` — марка кабеля (ППГнг(А)-HF)
- `phase_count` — количество фаз (обычно 4)
- `phase_section_mm2` — сечение фазной жилы
- `pe_or_n_count` — количество жил PE/N
- `pe_or_n_section_mm2` — сечение PE/N

**Как детектировать:**
```python
for line_id, cable in symbol_table["cables"].items():
    for field in ["phase_section", "pe_section", "pe_count", "mark"]:
        values_by_source = collect_field_values(cable, field)
        if len(set(values_by_source.values())) > 1:
            yield candidate(
                type="cable_attribute_mismatch",
                entity=line_id,
                field=field,
                values=values_by_source,
            )
```

**Критично:** extractor должен **не терять source** (где именно видел значение). Без этого judge не сможет проверить.

**Вес в эталоне:** 40% (2 из 5 KEEPs)

#### P1.3 — note_vs_equipment_conflict (Группа 3.1)
**Покрывает: #55 (ВА-335А vs примечание об электронных расцепителях)**

**Суть:** в примечаниях к листу декларировано требование к оборудованию, фактически установленное оборудование ему не соответствует.

**Типы правил которые умеем детектировать:**
- "применить АВ с электронными расцепителями" → проверка типа расцепителя каждого АВ
- "ТТ класса точности не хуже 0.5S" → проверка класса точности ТТ
- "кабели с индексом FR" → проверка марки кабелей
- "не менее 10% резервных АВ" → подсчёт занятых позиций
- "применить в огнестойком исполнении EI150" → проверка маркировки

**Как детектировать:**
```python
for note in symbol_table["notes"]:
    rule = parse_note_rule(note.text)  # LLM-helper или regex
    if rule is None:
        continue
    scope_entities = resolve_scope(rule.scope_description, symbol_table)
    for entity in scope_entities:
        if not entity_matches_rule(entity, rule):
            yield candidate(
                type="note_vs_equipment_conflict",
                rule_source=note,
                violating_entity=entity,
                rule_description=rule.requirement,
            )
```

**Сложность:** требует **парсинга текста примечаний** в структурированное правило. Это не чистый Python — нужен LLM-вызов для превращения «применить АВ с электронными расцепителями» в `{required_trip_type: "electronic", scope: "отходящие АВ ГРЩ"}`.

**Вес в эталоне:** 20% (1 из 5 KEEPs)

### Priority 2 — дополнительные классы для production

Эти классы дают ценность, но в текущем эталоне нет подтверждённых KEEPs. Реализация после того как P1 прототип докажет концепцию.

#### P2.1 — equipment_attribute_mismatch (Группа 1.2)
Расхождение марок/типов оборудования между источниками (шинопровод алюминиевый vs медный, КХА-II vs КХА, источник питания ЩСН РП1 vs ВРУ-1).

**В эталоне:** 5 unknown — #50, #51, #72, #73, #74. Если 2-3 подтвердятся — класс становится важным.

#### P2.2 — table_arithmetic_check (Группа 5 + 6)
Численные расчёты: потери напряжения ΔU > 5%, перегрузка ТТ (I_work > 1.2 × I_CT), коэффициент спроса Kс < 0.2 для крупной нагрузки, подсчёт сумм в таблицах.

**В эталоне:** 11 unknown в Группе 5 — потери напряжения (#35), cosφ (#41), ТТ 200/5А (#67), ТТ 30/5А (#68), Ics ВА-332А (#40). Это поле где Opus силён а Gemini/GPT слабы — потенциал для 3-5 новых KEEPs.

**Реализация:** **детерминированный Python** с формулами. LLM только извлекает числа, Python считает.

#### P2.3 — reserved_positions_check (Группа 3.3)
Проверка наличия минимум N% резервных позиций в щите как указано в примечаниях.

**В эталоне:** 1 неподтверждённое (F-008 из Opus прогона). Класс простой — **20 строк Python** + regex на поиск процента в примечаниях.

#### P2.4 — outdated_norms_reference (Группа 7)
Устаревшие ГОСТ/СП. **Важное правило от эксперта:** это `РЕКОМЕНДАТЕЛЬНОЕ` / `СПРАВОЧНО`, НЕ `КРИТИЧЕСКОЕ`.

**В эталоне:** 5 записей, все REMOVE с пометкой «нет материальной нагрузки». Это подтверждает правило.

**Реализация:** использовать существующий `norms_db.json`. Просто фильтр по severity.

#### P2.5 — power_category_check
Категория надёжности электроснабжения (I/II/III) должна быть указана в общих данных и соответствовать типу здания и составу нагрузок.

**В эталоне:** нет findings такого типа. Но эксперт подтвердил «обычно в общих данных указывается». Класс важен для проверки полноты документации.

**Реализация:** проверка наличия категории + сравнение с типом здания.

#### P2.6 — ukrm_presence_check
Наличие компенсации реактивной мощности (УКРМ) для объектов определённого типа.

**В эталоне:** 1 unknown (#15 УКРМ 20/25 vs 200/250 — 10x расхождение). Класс простой — проверка наличия и арифметики.

### Priority 3 — отключены или понижены

Эти классы генерируют больше шума чем сигнала, эксперт подтвердил низкую ценность.

#### P3.1 — phase_balance (Группа 5.1)
**Статус:** **ОТКЛЮЧЕН как автоматический finding**.

**Основание:** эксперт объяснил что перекос фаз на малой нагрузке с однофазным мощным потребителем — **физическое следствие**, а не ошибка проекта. Генератор без контекста будет давать FP.

**Условие включения:** только если нагрузка > 30 кВт И перекос > 50% И есть техническая возможность перераспределения (все потребители 1-фазные маломощные).

#### P3.2 — cable_trace_safety (Группа 4)
**Статус:** **ПОНИЖЕН приоритет**.

**Основание:** эксперт подтвердил «редко даёт реальные ошибки». Класс требует одновременного просмотра трёх типов чертежей (план трасс + узел разреза + примечания) — высокая сложность, низкая точность.

**Условие реализации:** только после P1 прототипа, с высоким порогом confidence.

#### P3.3 — text_quality (Группы 6, 8)
**Статус:** **ОТКЛЮЧЕНЫ**.

- Арифметические ошибки в OCR (#11, #16, #62) — OCR-артефакты
- Неверный адрес (#17), неверное название СП (#28), копипаста (#38) — все REMOVE

**Основание:** 100% FP в эталоне. Нужен не LLM, а более точный OCR или post-OCR валидация.

## Critical insights от эксперта (правила для judge-stage)

Эти правила — **обязательный фильтр** который применяется **после** генератора к любому кандидату:

### Rule 1 — Контекст для phase_balance
Не генерировать finding для перекоса фаз если:
- Общая нагрузка секции < 30 кВт
- Есть однофазный потребитель мощнее остальных в 5+ раз
- Все остальные потребители маломощные (< 2 кВт)

### Rule 2 — Severity устаревших норм
Устаревший ГОСТ/СП → максимум `РЕКОМЕНДАТЕЛЬНОЕ`. Никогда не `КРИТИЧЕСКОЕ`. Добавить комментарий «без материальной нагрузки».

### Rule 3 — Scope boundary check
Перед созданием finding проверить:
- Относится ли проблема к ГРЩ или к другому разделу (ЭМ, ЭО, ЭГ)?
- Если к другому разделу → НЕ finding (или `ПРОВЕРИТЬ ПО СМЕЖНЫМ` с низкой severity)

Примеры вне scope:
- Заземление (ЭГ)
- УЗДП (ЭМ)
- Молниезащита
- Селективность квартирных групповых сетей

### Rule 4 — Cable identification strictness
Для класса P1.2 (cable_attribute_mismatch): сопоставлять кабели **только по точному совпадению line_id** (M-1.1 == M-1.1). Никакого fuzzy matching по числам сечения — это источник ложных срабатываний.

### Rule 5 — OCR-подозрительность
Если расхождение — это одно из:
- Латиница vs кириллица в одинаковых марках (НФ vs HF)
- Одна цифра в 10x увеличении (1027 vs 1028, 648 vs 448)
- Частичная потеря символа в шифре (83 vs 133)

→ помечать как `likely_ocr_artifact` и **не поднимать** в финальные findings.

## Judge prompt template (минимальный)

```
Ты — эксперт-аудитор раздела ГРЩ (Главный распределительный щит) жилого здания.

Твоя задача: определить, является ли указанный ниже кандидат реальной ошибкой проекта.

## Кандидат

Тип: {candidate.type}
Сущность: {candidate.entity_id}
Поле/правило: {candidate.field_or_rule}

## Свидетельство A
Источник: {values[0].source_sheet} (стр. {values[0].source_page})
Блок: {values[0].source_block_id}
Значение: {values[0].value}
Цитата из документа: "{values[0].evidence_quote}"
[Изображение блока A]

## Свидетельство B
Источник: {values[1].source_sheet} (стр. {values[1].source_page})
Блок: {values[1].source_block_id}
Значение: {values[1].value}
Цитата из документа: "{values[1].evidence_quote}"
[Изображение блока B]

## Инструкции для анализа

1. Убедись что оба свидетельства описывают одну и ту же сущность (кабель, щит, автомат).
   Если это разные объекты → verdict = "invalid: different_entities"

2. Проверь не является ли расхождение OCR-артефактом:
   - Латиница vs кириллица в одинаковой марке → "invalid: ocr_artifact"
   - Одна цифра в большом числе (648 vs 448) → "invalid: ocr_artifact"
   - Потеря символа в шифре (83 vs 133) → "invalid: ocr_artifact"

3. Проверь что проблема относится к разделу ГРЩ (не к ЭГ/ЭМ/ЭО).

4. Если после фильтров расхождение реальное — verdict = "valid".

5. Если данных недостаточно — verdict = "uncertain".

## Выход (JSON)

{
  "verdict": "valid | invalid | uncertain",
  "reason_code": "ocr_artifact | different_entities | out_of_scope | real_conflict | ...",
  "explanation": "1-2 предложения",
  "corrected_value": "если можешь определить правильное значение"
}
```

## План реализации прототипа (3-4 часа)

### Шаг 1 — Extractor (1 час)
Новый промт для `block_analysis` на Gemini. Промт НЕ просит «найти ошибки», он просит **заполнить JSON schema** выше (entities.cables, entities.breakers, entities.panels, entities.notes).

Файл: `prompts/pipeline/en/block_extraction_task.md` (новый)
Модель: Gemini-3.1-pro (дёшево, умеет structured output)
Проверка: прогон на одном batch_001 ГРЩ, проверка JSON

### Шаг 2 — Symbol Table Builder (1 час)
Python-скрипт читает все `block_batch_*.json` (новый формат) + `01_text_analysis.json`, строит канонический symbol table.

Файл: `build_symbol_table.py`
Структура: dict с cables/breakers/panels/notes, каждый со списком mentions.

### Шаг 3 — Candidate Generators (1 час)
3 функции на Python, каждая 20-50 строк.

Файл: `candidate_generators.py`
- `gen_identity_uniqueness(symbol_table) → [candidates]`
- `gen_cable_attribute_mismatch(symbol_table) → [candidates]`
- `gen_note_vs_equipment_conflict(symbol_table) → [candidates]` (требует LLM для парсинга note)

### Шаг 4 — Judge + Finding formatter (30 мин)
Переиспользовать существующий findings_critic. Передавать кандидаты по одному.

### Шаг 5 — Прогон + оценка (30 мин)
На ГРЩ, сравнение с эталоном через `evaluate_against_etalon.py`.

### Ожидаемый результат
- **Если прототип ловит все 5 KEEPs** — доказательство концепции, идём в production
- **Если ловит 3-4** — понимаем на чём споткнулось, итерируем конкретный генератор
- **Если ловит 0-2** — значит проблема в extractor (не умеет извлекать достаточно полей), возвращаемся к промту extractor
