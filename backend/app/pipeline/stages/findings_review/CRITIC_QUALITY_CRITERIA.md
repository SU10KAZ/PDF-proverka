# Критерии качества замечаний (Findings Quality Criteria)

Этот документ описывает критерии, по которым critic оценивает каждое замечание.
Он служит опорным стандартом для fixtures, offline-тестов и будущего critic v2.

Источник: согласованные правила из `Experiments_Kuldyaev/RULES_DRAFT.md`
и контракт `Experiments_Kuldyaev/new_critic/CONTRACT.md`.

---

## Что делает critic

Critic проверяет каждое замечание в **два слоя**:

1. **Фактическая обоснованность** — подтверждён ли факт дефекта.
2. **Отчётная ценность** — стоит ли выводить это замечание в реестр.

Доказанный факт не всегда является полезным замечанием. Critic должен отделять
эти два вопроса.

---

## Критерии полезного замечания

Замечание считается полезным и должно получить `verdict: pass`, если:

| Критерий | Описание |
|---|---|
| **Конкретный объект** | Помещение, узел, оборудование, материал, трасса, лист, строка спецификации — не "проект в целом" |
| **Конкретная проблема** | Несоответствие, отсутствие, противоречие — не "требуется уточнить" |
| **Проверяемое последствие** | Выводится из нормы, расчёта, факта "декларация vs факт" или инженерной логики |
| **Evidence присутствует** | Заполнены `evidence[]` или `related_block_ids[]`, block_id существуют |
| **Практическая ось** | Хотя бы одна ось влияния: `construction`, `safety`, `acceptance`, `legal`, `reliability`, `operations`, `cost_schedule` |
| **Новая информация** | Не дублирует другое замечание с тем же объектом, причиной и исправлением |

### Оси практического влияния

- `construction` — влияет на монтаж, закупку, поставку, производство работ
- `safety` — пожарная, конструктивная, электрическая или иная безопасность
- `acceptance` — реальный риск предписания, невозможность проверки обязательного параметра
- `legal` — юридически значимый риск при производстве или сдаче работ
- `reliability` — надежность конструкций или инженерных систем
- `operations` — эксплуатация, обслуживание, доступ, ремонт
- `cost_schedule` — стоимость, сроки, переделки

## Критерии бесполезного замечания

Замечание не должно получить `verdict: pass`, если:

### 1. Общие слова без конкретики

Признаки:
- "Необходимо проверить X"
- "Рекомендуется уточнить Y"
- "Возможно, не соответствует"
- "Вероятно, требует пересмотра"
- "Следует рассмотреть вопрос"

Вердикт: `weak_evidence` или `not_practical`.

### 2. Нет evidence

Признаки:
- `evidence` пустой или отсутствует
- `related_block_ids` пустой или отсутствует

Вердикт: `no_evidence`.

### 3. Битые block_id (phantom blocks)

Признаки:
- block_id из `evidence` не существует в `02_blocks_analysis.json`

Если есть валидные block_id — `narrow_evidence`.
Если все block_id битые — `phantom_block` + `remove`.

### 4. Нет практического влияния

Признаки:
- Опечатка, не меняющая смысл
- Косметика ведомости
- OCR-артефакт
- Расхождение названия листа без потери смысла
- Ошибка адреса/шифра, если документ однозначно идентифицируется

Вердикт: `not_practical` + `report_value: remove`.

### 5. Параметрика без доказательства

Признаки:
- Утверждение "значение неверное" без расчёта
- Нет нормативного предела для сравнения
- Нет пары "декларация vs факт"

Вердикт: `weak_evidence` + `report_value: downgrade`.

### 6. Замечание-дубль

Признаки:
- Тот же объект
- Та же причина дефекта
- То же требуемое исправление

Вердикт: `not_practical` + `report_value: merge` + `merge_into: <ID>`.

Исключение: повтор полезен, если добавляет новый лист, зону, расчёт или evidence.

### 7. Замечания об отсутствии без проверки области

Признаки:
- Вывод "X отсутствует" сделан только по одному evidence-блоку
- Релевантная область не проверена

Вердикт: `weak_evidence` или `report_value: downgrade`.

## Строгий режим для категорий `documentation` и `normative_refs`

Эти категории не проходят только потому, что факт найден.

Для `pass` нужно объяснить, какое реальное действие изменится:
- монтаж
- закупка
- расчёт
- безопасность
- эксплуатация
- приемка обязательного требования
- юридически значимая исполнительная документация

Если такого влияния нет → `not_practical` + `report_value: remove`.

## Итоговые вердикты critic

| Вердикт | Значение |
|---|---|
| `pass` | Факт подтверждён, привязка корректна, есть отчётная ценность |
| `no_evidence` | Нет evidence и related_block_ids |
| `phantom_block` | block_id не существует в данных |
| `weak_evidence` | Evidence не подтверждает суть или параметрика не доказана |
| `page_mismatch` | Перепутаны page/sheet, суть может быть верной |
| `contradicts_text` | Замечание прямо противоречит тексту документа |
| `not_practical` | Факт может быть верным, но нет отчётной ценности |

## Новые аналитические поля (critic v2)

- `factual_verdict`: `supported | unsupported | contradicted | unclear`
- `report_value`: `include | merge | downgrade | remove`
- `object_problem_consequence`: `pass | fail | partial`
- `impact_axes`: массив осей влияния
- `practicality_reason`: краткое объяснение
- `merge_into`: ID основного замечания или `null`

1. При добавлении fixtures — проверяй, что `expected_decision` соответствует
   критериям выше.

3. При code review изменений critic — проверять соответствие.
