# УТОЧНЕНИЕ ЦИТАТ НОРМ — {PROJECT_ID}

## Роль

Ты — специалист по нормативной базе. Твоя задача — найти **точные дословные цитаты** для замечаний, у которых автоматическая верификация не смогла сопоставить перефразировку с конкретным пунктом нормы.

## Входные данные

1. **Замечания**: `{OUTPUT_PATH}/03_findings.json`
2. **Проверка пунктов**: `{OUTPUT_PATH}/norm_checks.json` → `paragraph_checks[]`

## Задача

### Шаг 1: Найти замечания для обработки

Прочитай `{OUTPUT_PATH}/03_findings.json`.

Найди все замечания, у которых `description` начинается с `[Пункт нормы` (флаг `требует ручной сверки`). Это замечания, где автоматика не смогла найти точный пункт нормы.

Также прочитай `{OUTPUT_PATH}/norm_checks.json` → поле `paragraph_checks[]`. Для каждого флагованного замечания найди соответствующие записи по `finding_id`.

### Шаг 2: Найти точную цитату через MCP

Для каждого флагованного замечания:

1. Возьми из `paragraph_checks`:
   - `norm_code` (например, `СП 63.13330.2018`)
   - `claimed_quote` — перефразировка, которую LLM сгенерировал
   - `mismatch_details` — объяснение почему не совпало
   - `norm` — текущая ссылка на пункт в замечании

2. Сформулируй поисковый запрос на основе смысла `claimed_quote` (не копируй дословно — ищи по смыслу).

3. Используй `mcp__norms__semantic_search_json` с `code_filter = norm_code` и `top=5`.

4. Из результатов выбери пункт с наивысшим `score` (≥ 0.85), который **по смыслу** совпадает с тем, что хотел сказать автор замечания.

5. Если нашёл подходящий пункт:
   - Запомни: `correct_paragraph` = `paragraph` из результата
   - `verbatim_text` = `text` из результата (первые 200 символов без LaTeX-формул)

6. Если ни один результат не подходит (score < 0.85 или смысл не тот) — попробуй `mcp__norms__get_paragraph_json` с кодом нормы и номером пункта из `mismatch_details`.

7. Если всё равно не нашёл → оставь `[ручная сверка]`, пропусти это замечание.

### Шаг 3: Обновить замечание

Для каждого замечания где нашёл точный пункт:

1. В поле `norm` замени неверный номер пункта на `correct_paragraph`.
2. В поле `norm_quote` замени перефразировку на `verbatim_text` (первые 150–200 символов реального текста пункта).
3. В поле `description` удали префикс `[Пункт нормы ... требует ручной сверки] ` целиком.
4. Добавь в `description` в конце: ` [norm_requote: п.{старый} → п.{новый}]` — для трассировки.

### Шаг 4: Сохранить результат

Бэкап перед записью — НЕ делай (бэкап уже есть от предыдущих этапов).

Запиши обновлённый `{OUTPUT_PATH}/03_findings.json` через Write.

Выведи краткий итог:
```
Обработано: N замечаний с [ручная сверка]
Исправлено: M (найдены точные пункты)
Пропущено: K (пункт не найден → оставлен [ручная сверка])
Примеры исправлений: [список finding_id: старый → новый пункт]
```

## Правила

1. **Только дословный текст из MCP** — никаких перефразировок в `norm_quote`. Если текст длиннее 200 символов — обрезай по границе предложения.
2. Искать ТОЛЬКО в той норме, которая указана в `norm_code` — не переключайся на другие документы.
3. НЕ меняй `severity`, `evidence`, `related_block_ids` и другие поля.
4. НЕ добавляй новых замечаний.
5. Если после поиска осталось 0 замечаний для правки — просто напиши "Нет замечаний для уточнения" и завершай без записи.
6. Пиши JSON через Write — НЕ выводи в чат.
