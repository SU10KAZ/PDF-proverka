## Входные данные

1. **Контекст текстового анализа**: `{OUTPUT_PATH}/01_text_analysis.json`
   - `text_findings` — замечания из текста (для верификации на чертежах)
   - `project_params` — параметры проекта (нагрузки, расходы, марки)
   Если файл не существует — работать без контекста.

2. **Структурированный контекст страниц** (из Document Knowledge Graph):

{BLOCK_MD_CONTEXT}

   Контекст организован per-block:
   - **Метаданные страницы** (Лист, Наименование) — координаты чертежа
   - **Текст на странице** — таблицы, примечания, спецификации, условные обозначения (общий для всех блоков страницы)
   - **Per-block OCR** — OCR-описание конкретного блока (для каждого блока пакета)
   - **Другие блоки на странице** — список block_id соседних блоков (для межблочной сверки)

   **ВАЖНО:** Сверяй данные на чертеже (PNG) с текстом на той же странице.
   Расхождения между чертежом и текстом на той же странице → замечание.

3. **Блоки для анализа** (прочитать КАЖДЫЙ через Read):
{BLOCK_LIST}

Каждый блок — кропнутый фрагмент чертежа (законченная область: схема, план, таблица или узел). Анализируй каждый блок как самостоятельный чертёж.

**ВАЖНО: page vs sheet (Лист).** `page` = номер страницы в PDF-файле (физическая нумерация). `sheet` = номер листа в составе комплекта чертежей (логическая нумерация из штампа). Они НЕ совпадают: Лист 1 может быть на стр. PDF 5. В списке блоков выше указаны оба. В `sheet` записывай номер листа из штампа или из контекста страницы (`**Лист:**`).

### СТОП-ПРОВЕРКА page vs sheet (перед записью КАЖДОГО блока)

```
page = число из списка блоков выше (после "page:")     ← ФИЗИЧЕСКАЯ страница PDF
sheet = значение из контекста "**Лист:** Лист N"        ← ЛОГИЧЕСКИЙ лист из штампа
```

**ТИПИЧНАЯ ОШИБКА:** page=7, контекст говорит "**Лист:** Лист 3" → sheet="Лист 3" (НЕ "Лист 7"!)

- Если `**Лист:**` указан в контексте → бери sheet оттуда
- Если `**Лист:**` НЕ указан → `sheet: null` (НЕ копируй page в sheet, НЕ угадывай)

## ОДНОЛИНЕЙНЫЕ И РАСЧЁТНЫЕ СХЕМЫ — ПОЛНОЕ РАСПОЗНАВАНИЕ ТЕКСТА (ОБЯЗАТЕЛЬНО)

Если блок содержит однолинейную схему, расчётную схему, схему ВРУ/ГРЩ/УЭРМ/щита:

**ОБЯЗАТЕЛЬНО прочитай и запиши в `key_values_read` ВСЕ надписи на схеме, включая:**
1. **Вертикальный текст вдоль кабельных линий** — марки и сечения кабелей (ВВГнг(А)-FRLS 5x10, ВВГнг-LS 3x2.5 и т.д.)
2. **Длины кабелей** — значения в метрах рядом с линиями (L=25м, 48м и т.д.)
3. **Марки автоматов и УЗО** — номиналы, типы (ВА47-29 С16, АВВ S203 и т.д.)
4. **Токи трансформаторов тока** — ТТ (200/5А, 400/5А и т.д.)
5. **Расчётные токи и мощности** — Ip, Pp, Sp, cosφ у каждого фидера
6. **Позиционные обозначения** — QF1, QS1, KM1, TA1 и т.д.
7. **Надписи у шин** — маркировки шин, секций, вводов

**НЕ ПРОПУСКАЙ текст только потому, что он мелкий или повёрнут.** Блоки однолинейных схем отрендерены в повышенном разрешении (2500px) специально для полного распознавания.

Если текст нечитаем — запиши "нечитаемо: [описание где именно]" в `key_values_read`.

**ВАЖНО — обратная связь по нечитаемому тексту:**
Если на блоке есть текст, который ты не можешь прочитать из-за низкого разрешения (мелкий шрифт, таблицы с цифрами, марки кабелей, номиналы автоматов), установи поле `unreadable_text: true` и заполни `unreadable_details` — опиши ГДЕ именно и ЧТО не читается. Система автоматически перекачает этот блок в повышенном разрешении и повторит анализ.
Если весь текст читаем — ставь `unreadable_text: false`.

## Сверка с текстовым анализом (ОБЯЗАТЕЛЬНО)

Из `01_text_analysis.json` → `project_params` извлечь числовые данные.
На КАЖДОМ чертеже сверять видимые значения с данными из текста:
- Расходы, нагрузки, мощности — совпадают с таблицами?
- Диаметры, сечения — совпадают со спецификацией?
- Типоразмеры оборудования — существуют в каталогах?
- **Марки кабелей** — совпадают ли с позициями спецификации?
- **Длины кабелей** — соответствуют ли кабельному журналу?

Любое расхождение → замечание в findings.

**ВАЖНО:** Если не можешь уверенно прочитать значение на чертеже (мелкий текст, низкое разрешение, перекрытие) — пиши "нечитаемо" в `key_values_read`. НЕ угадывай цифры и марки.

### Поля locality (ОБЯЗАТЕЛЬНЫ для каждого block_analysis):

- **`evidence_text_refs`** — трассировка: для каждого использованного text-блока укажи:
  - `text_block_id` — ID блока
  - `role` — роль text-блока: `"caption"` (подпись), `"note"` (примечание), `"legend"` (условные обозначения), `"title"` (заголовок), `"table"` (таблица/спецификация), `"other"`
  - `used_for` — для чего использован: `"summary"` (описание), `"finding"` (замечание), `"value_extraction"` (извлечение значений), `"cross_check"` (перекрёстная сверка)

Если text-контекст не содержал `[text_block_id: ...]` меток, ставь `"evidence_text_refs": []`.

## Визуальная привязка замечаний (highlight_regions) — ОБЯЗАТЕЛЬНО

**КАЖДОЕ замечание ОБЯЗАНО иметь непустой массив `highlight_regions`.** Это используется для показа пользователю где именно на чертеже находится проблема.

Координаты — **нормализованные** (0.0–1.0) относительно размера блока:
- `x`, `y` — верхний левый угол области (доля от ширины/высоты блока)
- `w`, `h` — ширина и высота области (доля от ширины/высоты блока)
- `label` — краткое пояснение (название оборудования, марка кабеля, размер и т.д.)

**Быстрая таблица координат (выбери ближайший квадрант):**

| Расположение на чертеже | x | y | Типичная w | Типичная h |
|-------------------------|-----|-----|------------|------------|
| Верхний левый угол | 0.0 | 0.0 | 0.3 | 0.3 |
| Верх по центру | 0.3 | 0.0 | 0.4 | 0.3 |
| Верхний правый угол | 0.7 | 0.0 | 0.3 | 0.3 |
| Середина слева | 0.0 | 0.3 | 0.3 | 0.4 |
| Центр | 0.3 | 0.3 | 0.4 | 0.4 |
| Середина справа | 0.7 | 0.3 | 0.3 | 0.4 |
| Нижний левый угол | 0.0 | 0.7 | 0.3 | 0.3 |
| Низ по центру | 0.3 | 0.7 | 0.4 | 0.3 |
| Нижний правый угол | 0.7 | 0.7 | 0.3 | 0.3 |

**Правила:**
1. Для конкретного элемента (кабель, автомат, воздуховод, надпись) — узкий прямоугольник вокруг него
2. Для проблемы в таблице/спецификации — прямоугольник вокруг соответствующей области таблицы
3. Для отсутствующего элемента — прямоугольник вокруг области, где он ДОЛЖЕН быть
4. Несколько областей допускается (например, два конфликтующих значения в разных местах)
5. **Fallback: если замечание касается всего чертежа** → `[{{"x": 0, "y": 0, "w": 1, "h": 1, "label": "Весь чертёж — <причина>"}}]`

**НИКОГДА не возвращай пустой `highlight_regions: []`.** Если не можешь точно определить область — используй fallback на весь блок.

## Критерии для «ПРОВЕРИТЬ ПО СМЕЖНЫМ»

**СТАВИТЬ** только при конкретном противоречии: числовом расхождении (размер на чертеже vs расчётный), нормативном конфликте (категория/класс), физической несовместимости (оборудование не помещается).

**НЕ СТАВИТЬ** если на чертеже просто видны элементы смежных дисциплин (электрощит, трубы, вентрешётки) — это нормальная координация, не ошибка.

## Правила

1. Прочитай КАЖДЫЙ блок — не пропускай ни один
2. Для каждого блока ОБЯЗАТЕЛЬНО: summary и key_values_read
3. Штампы → summary: "Штамп / служебная информация"
4. findings может быть пустым [] если замечаний нет
5. Нумерация: G-001, G-002... (в пределах пакета)
6. severity — ТОЛЬКО одно из 5 значений
7. Пиши JSON через Write — НЕ выводи в чат
8. После записи выведи краткий итог

## Нормативная точность (norm_quote)

Для КАЖДОГО замечания с полем `norm`:
- **`norm_quote`** — точная цитата текста из нормативного документа (1-2 предложения), на которую опирается замечание. Если не помнишь точную формулировку — поставь `null`.

Все цитаты будут проверены на этапе верификации норм (этап 04) независимо от уверенности.

## Нормативная база
Нормативная проверка выполняется на этапе свода замечаний — здесь фиксируй только факты с чертежей.
