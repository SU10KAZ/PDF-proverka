# ЗАДАЧА АУДИТА — ИНТЕЛЛЕКТУАЛЬНЫЙ КОНВЕЙЕР (SMART PIPELINE)

## Режим работы
Работай АВТОНОМНО. Не задавай вопросов. Не жди подтверждений.
Каждый этап ОБЯЗАН записать JSON-файл перед переходом к следующему.
Следующий этап читает JSON предыдущего — НЕ сканирует контекст заново.

## Структура конвейера

```
СТАРТ
  │
  ▼
[00] Инициализация ──────────────────────► ПИШЕТ: _output/00_init.json
  │   Подготовка текста (без нарезки тайлов!)
  ▼
[01] Анализ текста + ТРИАЖ СТРАНИЦ ──────► ПИШЕТ: _output/01_text_analysis.json
  │   ЧИТАЕТ: MD-файл / extracted_text.txt      (включает page_triage[])
  │   Определяет какие чертежи проверять
  ▼
[01.5] Выборочная нарезка тайлов ────────► Только HIGH+MEDIUM страницы
  │   python process_project.py --pages X,Y,Z
  ▼
[02] Анализ выбранных тайлов ────────────► ПИШЕТ: _output/02_tiles_analysis.json
  │   ЧИТАЕТ: 01_text_analysis.json              (iteration: 1)
  ▼
[02.5] Анализ пробелов ─────────────────► Нужны ещё чертежи?
  │   ЕСЛИ ДА → нарезать доп. страницы,
  │             проанализировать, обновить 02_tiles_analysis.json
  │             (iteration: 2, макс. 3 итерации)
  ▼
[03] Свод замечаний ─────────────────────► ПИШЕТ: _output/03_findings.json
  │   ЧИТАЕТ: 01_text_analysis.json + 02_tiles_analysis.json
  ▼
[04] Финальный отчёт ───────────────────► ПИШЕТ: _output/audit_results_YYYYMMDD.md
      ЧИТАЕТ: 03_findings.json
КОНЕЦ
```

ПРАВИЛО: ВСЕГДА выполняй все этапы. Перезаписывай существующие JSON-файлы.
Каждый запуск аудита должен создавать СВЕЖИЕ результаты на основе актуальных данных.

### Шаг 01.1: Выбор текстового источника

**MD-файл:** `{MD_FILE_PATH}`

Если путь к MD-файлу НЕ "(нет)":
1. **ЧИТАТЬ MD-файл как ОСНОВНОЙ текстовый источник** — он содержит:
   - Структурированный текст каждой страницы (`## СТРАНИЦА N`)
   - Таблицы в формате Markdown (спецификации, ведомости — точнее чем PyMuPDF)
   - Описания чертежей (блоки `IMAGE:` — тип, оси, сущности)
   - ~2x больше информации чем extracted_text.txt
2. `extracted_text.txt` использовать как **дополнительный источник** для перекрёстной проверки
3. При расхождениях между MD и extracted_text.txt — тайлы (PDF) покажут истину

Если MD-файл отсутствует ("(нет)"):
- Читать `_output/extracted_text.txt` как единственный текстовый источник

## ЭТАП 01.5 — ВЫБОРОЧНАЯ НАРЕЗКА ТАЙЛОВ

**НОВЫЙ ЭТАП.** Нарезка тайлов ТОЛЬКО для приоритетных страниц.

### Шаг 01.5.1: Определить страницы для нарезки

Из `01_text_analysis.json` → `page_triage`:
- **Раунд 1:** все страницы с priority = HIGH + MEDIUM

## ЭТАП 02.5 — АНАЛИЗ ПРОБЕЛОВ (GAP ANALYSIS)

**НОВЫЙ ЭТАП.** Определить, достаточно ли данных для полноценного аудита.

### Шаг 02.5.1: Проверить пробелы

Прочитать `02_tiles_analysis.json` и проверить:

1. **Ссылки на другие чертежи:** Найдены ли замечания, которые требуют данных
   с чертежей, которые ещё НЕ были проанализированы?
   Пример: F-005 говорит "сечение не соответствует спецификации" → нужна страница спецификации

2. **Неразрешённые verify_tile:** Есть ли в `01_text_analysis.json` находки с
   `needs_tile_verification: true`, для которых тайлы ещё не были нарезаны?

3. **Подозрительные данные:** Замечены ли на проанализированных чертежах ссылки
   на листы/щиты, которые не были проверены?

### Шаг 02.5.2: Решение

**Если пробелов НЕТ** (или оставшиеся страницы LOW/SKIP):
- Записать `gap_analysis: null` в 02_tiles_analysis.json
- Перейти к этапу 03

**Если нужны дополнительные страницы:**

1. Обновить `02_tiles_analysis.json` с gap_analysis:
```json
"gap_analysis": {
  "iteration": 1,
  "additional_pages_needed": [12, 14],
  "reasons": [
    {"page": 12, "reason": "Замечание F-005 ссылается на спецификацию — нужна проверка объёмов"},
    {"page": 14, "reason": "Узел ввода — проверить сечения вводных кабелей по Iр=145А"}
  ],
  "pages_analyzed_so_far": [7, 9, 11]
}
```

2. Нарезать дополнительные тайлы:
```bash
python process_project.py projects/<name> --pages 12,14 --quality speed
```

3. Прочитать новые тайлы и дополнить `02_tiles_analysis.json`:
   - Добавить новые записи в `tiles_reviewed`
   - Добавить новые `preliminary_findings`
   - Обновить `meta.iteration` → 2
   - Обновить `meta.pages_analyzed` → [7, 9, 11, 12, 14]

4. **ОГРАНИЧЕНИЕ:** максимум 3 итерации. После 3-й итерации — переходить к этапу 03
   вне зависимости от оставшихся пробелов.

## ПРАВИЛА JSON

1. Каждый JSON писать через инструмент Write — не выводить в чат
2. Формат даты: ISO 8601 — "2026-02-26T23:55:00"
3. ID: T-001..999 (текст), G-001..999 (чертежи), F-001..999 (финал)
4. При ответах на вопросы — читать 03_findings.json, не весь контекст
5. Максимум 3 итерации нарезки+анализа тайлов (защита от бесконечного цикла)

## Входные данные
Проекты находятся в папке: `projects/`

Для каждого проекта:
- `projects/<name>/document.pdf`               — оригинал (источник истины)
- `projects/<name>/<name>_document.md`         — структурированный текст (Chandra)
- `projects/<name>/_output/extracted_text.txt` — текст PDF (fallback)
- `{DISCIPLINE_NORMS_FILE}`                    — нормативная база дисциплины

## Задача
Выполни полный технический аудит КАЖДОГО проекта из папки `projects/`.

### Шаги для КАЖДОГО проекта (без пропусков):

**1. ИНИЦИАЛИЗАЦИЯ**
- Прочитать `projects/<name>/project_info.json`
- Проверить наличие текста (MD или extracted_text.txt)
- Записать 00_init.json

**2. ТЕКСТОВЫЙ АНАЛИЗ + ТРИАЖ**
- Прочитать MD-файл (или extracted_text.txt)
- Найти нормативные ссылки, расчётные параметры, кабели
- Определить page_triage — какие чертежи нужно проверять
- Записать 01_text_analysis.json

**3. ВЫБОРОЧНАЯ НАРЕЗКА ТАЙЛОВ**
- Нарезать только HIGH + MEDIUM страницы из page_triage
- `python process_project.py projects/<name> --pages X,Y,Z --quality speed`

**4. АНАЛИЗ ЧЕРТЕЖЕЙ**
- Прочитать тайлы нарезанных страниц
- Найти проблемы на чертежах
- Записать 02_tiles_analysis.json

**5. АНАЛИЗ ПРОБЕЛОВ** (если нужно)
- Проверить: нужны ли дополнительные чертежи?
- Если да — нарезать и проанализировать (макс. 3 итерации)

**6. ФОРМИРОВАНИЕ ЗАМЕЧАНИЙ**
- Объединить находки текстового и визуального анализа
- Записать 03_findings.json

**7. ФИНАЛЬНЫЙ ОТЧЁТ**
- Записать audit_results_YYYYMMDD.md

## Тип вывода
- Отдельный файл audit_results_*.md для каждого проекта
- В конце — общее резюме по всем проектам
