## Загруженные данные

**MD-файл** — текстовая версия проекта (основной рабочий файл). Используй его для анализа: поиск цифр, сравнение таблиц, расчёты, проверка обозначений, изучение спецификаций. Весь текст здесь точный и доступен для поиска.

**Тайлы** — фрагменты PDF-чертежей в `{OUTPUT_PATH}/tiles/`. Используй для визуальной сверки: принципиальные схемы, узлы, разрезы, планы, расположение элементов, связи и габариты.

**02_tiles_analysis.json** — готовый анализ тайлов (если есть). Читай его вместо прямого просмотра всех тайлов.

**ПРАВИЛО:** Если данные в MD и тайлах расходятся — верь тайлам (PDF-оригинал).

## Направления оптимизации

### 1. ЗАМЕНА НА БОЛЕЕ ДЕШЁВЫЕ АНАЛОГИ
Проанализируй спецификации и ведомости оборудования/материалов. Предложи аналоги с сопоставимыми техническими характеристиками, но по более низкой цене. Для каждой замены укажи: исходную позицию, предлагаемый аналог, ориентировочную разницу в цене (%), обоснование эквивалентности (параметры, сертификаты, совместимость, ресурс).

### 2. ЗАМЕНА МАТЕРИАЛОВ/ОБОРУДОВАНИЯ ДЛЯ УСКОРЕНИЯ ИЛИ УДЕШЕВЛЕНИЯ МОНТАЖА
Найди позиции, где выбор материала или оборудования усложняет монтаж. Предложи замены, сокращающие трудозатраты (более технологичные материалы, переход со сварки на сборные соединения, быстромонтируемые крепежи и т.д.). Указывай экономию и в стоимости материала, и в человеко-часах.

### 3. ИЗМЕНЕНИЕ УЗЛОВ СБОРКИ И КОНСТРУКТИВА ДЛЯ УДЕШЕВЛЕНИЯ РАБОТ
Оцени компоновочные, конструктивные и сборочные решения (обвязки, коллекторы, щиты, крепежные конструкции, проходки, архитектурные узлы). Предложи упрощение конструкции, унификацию деталей, сокращение количества соединений или переход на готовые заводские сборки (модули) вместо ручной сборки на стройплощадке.

### 4. ОПТИМИЗАЦИЯ, ВЛИЯЮЩАЯ НА ЭКСПЛУАТАЦИЮ И ЖИЗНЕННЫЙ ЦИКЛ
Предложи решения, которые могут быть дороже на этапе строительства, но дадут экономию на горизонте 10-25 лет: энергоэффективность, снижение тепловых или гидравлических потерь, повышение ремонтопригодности, увеличение межсервисных интервалов, доступность и снижение стоимости ЗИП.

## Правила анализа

- Все замены должны строго соответствовать действующим строительным нормам (СП, ГОСТ, СНиП, ТР ТС).
- **НЕ** предлагать замены, снижающие надёжность, несущую способность, пропускную способность или пожарную безопасность.
- При сомнениях — помечать позицию как `"status": "требует проверки"`.
- Приоритет: сначала крупные позиции (оборудование, основные материалы) с максимальным экономическим эффектом, затем мелкие.
- Если обнаружены расхождения между MD и тайлами — указать в поле `risks`.

## Порядок работы

1. Прочитай MD-файл целиком — сформируй понимание проекта (тип объекта, ТЭП, нагрузки/мощности, основное оборудование).
2. Прочитай `02_tiles_analysis.json` (если есть) для визуальных данных.
3. Проанализируй спецификации и ведомости — это основной источник позиций для оптимизации.
4. Сформируй оптимизационный лист по направлениям 1-4.
5. Запиши результат в `{OUTPUT_PATH}/optimization.json`.

## Формат вывода

Запиши результат в файл `{OUTPUT_PATH}/optimization.json`.

### Правила заполнения полей

| Поле | Тип | Описание |
|------|-----|----------|
| `id` | string | `"OPT-001"`, `"OPT-002"`, ... — порядковый номер |
| `section` | string | Откуда взята позиция: `"Спецификация лист 16"`, `"Схема ДВ2 лист 22"`, `"План 3 этажа лист 9"` |
| `current` | string | **ОДНА СТРОКА** — что есть в проекте сейчас (модель, параметры, проблема). Максимум 200 символов. |
| `proposed` | string | **ОДНА СТРОКА** — что предлагается взамен (модель, параметры, преимущество). Максимум 200 символов. |
| `type` | string | **ТОЛЬКО** одно из 4 значений: `"cheaper_analog"`, `"faster_install"`, `"simpler_design"`, `"lifecycle"` |
| `savings_pct` | number | Числовое значение экономии в процентах (0-100). Если экономия не числовая (исправление ошибки) — поставь `0`. |
| `timeline_impact` | string | `"без изменений"`, `"+2 дня"`, `"-3 дня"` и т.п. |
| `risks` | string | Одна строка с описанием рисков. Если рисков нет — `"минимальные"`. |
| `status` | string | `"предложение"`, `"требует проверки"` или `"обязательное исправление"` (для нарушений ПБ) |
| `norm` | string | Нормативная ссылка: `"СП 7.13130.2013 п. 7.11"`. Если не применимо — `""` |

### Типы оптимизации (`type`)

- `cheaper_analog` — замена на более дешёвый аналог
- `faster_install` — удешевление/ускорение монтажа
- `simpler_design` — упрощение конструктива / исправление ошибок проектирования
- `lifecycle` — оптимизация жизненного цикла

### Запреты

- **НЕ** создавай вложенные объекты (`current_equipment: {...}`) — только плоские строки
- **НЕ** переименовывай поля (`direction` вместо `type`, `estimated_saving` вместо `savings_pct`)
- **НЕ** добавляй поля, которых нет в схеме (`title`, `priority`, `action_required`)
- **НЕ** используй `null` — используй пустую строку `""` или `0`
- **НЕ** создавай MD-отчёт. Файл `optimization.json` — единственный выход.
