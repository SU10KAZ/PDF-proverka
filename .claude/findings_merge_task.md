## Задача

### Шаг 1: Межстраничная и межблочная сверка (ОБЯЗАТЕЛЬНО)

Перед объединением замечаний — сгруппируй `block_analyses[]` по страницам и проверь:

1. **Внутри одной страницы:**
   - Блоки одной страницы описывают одну концепцию (узлы крепления, каталожные листы и т.д.)
   - Нет ли противоречий между блоками одной страницы (разные размеры, марки, параметры)?
   - Однотипные блоки (например, 10 узлов крепления) — одинаковы ли принципы?

2. **Дисциплинарная фильтрация:**
   - Блоки с `discipline_mismatch: true` → замечания переносить как есть с severity "ПРОВЕРИТЬ ПО СМЕЖНЫМ"
   - НЕ повышать severity таких замечаний при объединении
   - НЕ проводить нормативную сверку по ним
   - Подсчитать количество таких блоков для `meta.cross_discipline_blocks`

3. **Между страницами:**
   - Спецификация (текст) vs чертежи: все ли позиции спецификации видны на чертежах? Есть ли на чертежах оборудование, отсутствующее в спецификации?
   - Каталожные листы (графики, характеристики) vs реально применённое оборудование: нет ли лишних каталожных листов для неприменяемых типоразмеров?
   - Параметры на одном чертеже (расход, диаметр) vs параметры на другом (таблица нагрузок, аксонометрия) — совпадают?
   - key_values_read с разных страниц — нет ли конфликтов?

Любое найденное расхождение → добавить как новое замечание (F-NNN).

### Шаг 2: Объединение замечаний

Объединить замечания из обоих этапов (01 текст + 02 блоки).

### Обработка items_verified_from_stage_01 (ОБЯЗАТЕЛЬНО)

Перед объединением — пройди по `items_verified_from_stage_01`:

- **`confirmed: true`** → текстовое замечание подтверждено чертежом. Повысить severity на одну ступень (РЕКОМЕНДАТЕЛЬНОЕ → ЭКСПЛУАТАЦИОННОЕ, ЭКСПЛУАТАЦИОННОЕ → ЭКОНОМИЧЕСКОЕ). КРИТИЧЕСКОЕ оставить как есть.
- **`confirmed: false`** → на чертеже видно другое, чем в тексте. Два варианта:
  - Если ошибка в тексте (опечатка, устаревшие данные), а на чертеже правильно → **убрать замечание** или понизить до РЕКОМЕНДАТЕЛЬНОЕ с пометкой "расхождение текста и чертежа"
  - Если на чертеже тоже ошибка, но другая → **оставить и уточнить** описание
- **Замечание без верификации** (T-NNN нет в items_verified) → оставить как есть, не повышать severity

### Правила объединения

1. **Дедупликация**: одно замечание найдено и в тексте, и на чертеже → одна запись с более полным описанием
2. **Повышение severity**: текстовое замечание подтверждено чертежом → severity повышается (см. items_verified выше)
3. **Понижение severity**: подозрение из текста НЕ подтвердилось на чертеже → понизить или убрать
4. **Перенумерация**: финальные ID: F-001, F-002...
5. **Привязка к блокам**: для каждого F-NNN заполни `related_block_ids` — список block_id из `02_blocks_analysis.json`, которые являются источником замечания. Для замечаний из G-NNN — `block_id` блока. Для T-NNN — block_id блоков, подтвердивших текстовое замечание (из `items_verified`). Для межблочных — все block_id участвующих блоков.

### Поля замечания

- `severity`: КРИТИЧЕСКОЕ / ЭКОНОМИЧЕСКОЕ / ЭКСПЛУАТАЦИОННОЕ / РЕКОМЕНДАТЕЛЬНОЕ / ПРОВЕРИТЬ ПО СМЕЖНЫМ
- `problem`: краткая суть (1-2 строки)
- `description`: развёрнутое описание с числовыми данными
- `norm`: документ + пункт (со статусом актуальности)
- `solution`: конкретное действие по исправлению
- `risk`: последствия если не исправить
- `merge_source_g_ids`: список ID исходных замечаний (G-NNN и/или T-NNN), из которых сформировано данное F-NNN. Для замечания из одного G → `["G-001"]`. Для объединённого T+G → `["T-003", "G-017"]`. Для нового (межблочная сверка) → `[]`.
- `source_block_ids`: список block_id, ГДЕ замечание реально ОБНАРУЖЕНО (source-of-truth). Это block_id блоков, на которых LLM увидела проблему. Отличается от `related_block_ids`: source = "где нашли", related = "к чему относится".
- `related_block_ids`: список block_id, К КОТОРЫМ замечание ОТНОСИТСЯ. Может включать блоки, где проблема не видна напрямую, но которые связаны (спецификация, каталожный лист и т.д.)
- `selected_text_block_ids`: ID text-блоков (из контекста `[text_block_id: ...]` в блочном анализе), использованных как evidence. Если в `blocks_compact` есть `selected_text_block_ids` для source block — перенеси сюда.
- `evidence_text_refs`: детальная трассировка text↔finding. Если в `blocks_compact` есть `evidence_text_refs` — перенеси и дедуплицируй.
- `evidence`: массив источников данных. `{type: "image"|"text", block_id: "...", page: N}`. type=image для графических блоков, type=text для текстовых блоков.
- `highlight_regions`: массив визуальных областей на блоке, где обнаружена проблема. Переноси из G-замечаний `02_blocks_analysis.json` → `findings[].highlight_regions`. Формат: `[{block_id: "...", x: 0.35, y: 0.40, w: 0.20, h: 0.15, label: "..."}]`. При объединении G-замечаний — собери все regions. Добавь `block_id` в каждый region чтобы знать к какому блоку относится область.

### Правила заполнения sheet и page (ОБЯЗАТЕЛЬНО)

- `sheet` — номер листа **из штампа** чертежа (поле `sheet_no` из контекста страницы / `02_blocks_analysis`). Формат: "Лист 7" или "Листы 3, 5". НЕ путать с номером страницы PDF!
- `page` — номер страницы PDF (целое число). Если замечание охватывает несколько страниц — массив `[12, 13]`.
- Лист из штампа и страница PDF НЕ совпадают: Лист 1 может быть на стр. PDF 5.

**ЖЁСТКОЕ ПРАВИЛО:** Если в compact input есть `page_sheet_map` — используй ТОЛЬКО его для определения sheet по page. Не выводи sheet по догадке или памяти. Если page_sheet_map говорит, что page 12 = Лист 7, пиши "Лист 7". Если page нет в page_sheet_map — поставь `"sheet": null` и НЕ угадывай.

## Нормативная точность (norm_quote + norm_confidence)

При объединении замечаний T-NNN и G-NNN в финальные F-NNN — **сохраняй** поля `norm_quote` и `norm_confidence` из исходных этапов.

Если два замечания объединяются в одно:
- `norm_quote` — берём из более подробного источника
- `norm_confidence` — берём минимальное из двух (более осторожная оценка)

Для новых замечаний (обнаруженных при межблочной сверке):
- Заполняй `norm_quote` и `norm_confidence` по тем же правилам

## Правила

1. Пиши JSON через Write — НЕ выводи в чат
2. ID замечаний: F-001, F-002... (сквозная нумерация)
3. При ссылке на норму — указывай статус (действует/заменён/отменён)
4. После записи выведи краткий итог: сколько замечаний, по критичности
